# goapp — Go backend template

An opinionated, production-shaped starting point for Go HTTP services. It ports
the **layered, operation-oriented architecture** of a mature FastAPI codebase to
idiomatic Go, and ships with everything a new service needs on day one:

- **Docker-first**: `docker compose up` brings up PostgreSQL, runs migrations and
  starts the API.
- **PostgreSQL** via [pgx](https://github.com/jackc/pgx) + type-safe queries
  generated by [sqlc](https://sqlc.dev) (no ORM runtime overhead).
- **Authentication**: classic JWT — short-lived access tokens + rotating refresh
  tokens (stored hashed for revocation), passwords hashed with bcrypt.
- **Authorization**: a Pundit-style **policy layer** combined with **RBAC** roles,
  with per-record permissions surfaced to the frontend.
- **Architecture guardrails** enforced as tests (pure decision layers, services
  that never authorize).

> The module path is `github.com/yourorg/goapp`. Rename it for your project:
> ```bash
> grep -rl github.com/yourorg/goapp . | xargs sed -i '' 's#github.com/yourorg/goapp#github.com/acme/myservice#g'
> # then: go mod edit -module github.com/acme/myservice && go mod tidy
> ```

## Tech stack & why

| Concern | Choice | Why |
|---|---|---|
| Router | stdlib `net/http` + [chi](https://github.com/go-chi/chi) | Idiomatic, no framework lock-in; the Mat Ryer / Grafana "`NewServer` + `routes`" pattern. |
| DB driver | pgx/v5 | Fastest, most complete PostgreSQL driver. |
| Queries | sqlc | Write SQL, get type-safe Go. Zero runtime reflection. Clean read/write split. |
| Migrations | [goose](https://github.com/pressly/goose) | Simple, plain-SQL, runs as a one-shot container. |
| Auth | golang-jwt/v5 + bcrypt | Standard, well-audited. |
| Config | env vars (+ `.env` in dev) | 12-factor; one typed `Config` struct. |
| Logging | stdlib `log/slog` | Structured, no dependency. |
| DI | manual constructor injection | Explicit, no magic — best for a template. |

## Architecture

The service is split into layers with strict, one-directional dependencies.

```
cmd/api                 process entrypoint (tiny main → app.Run)
internal/
  app/                  wiring: NewServer(deps) http.Handler + Run(ctx)
  platform/             cross-cutting platform support
    config/             typed configuration from the environment
    database/           pgx pool + sqlc-generated code (gen/)
    logger/             slog setup
    httpx/              response envelopes, decode/validate, error rendering
    middleware/         recover, request logging, CORS
    authz/              Actor, RBAC roles, Authorize(), auth middleware
  auth/                 authentication: JWT, password, login/refresh/logout/me
  user/                 user domain: model, read/write, policy, dto
  article/              example domain (full pattern, see below)
pkg/
  apperror/             the one application error type (status + message_key)
db/
  migrations/           goose SQL migrations (source of truth for the schema)
  queries/              sqlc query definitions
```

### Layer responsibilities

| Layer | Does | Must **not** |
|---|---|---|
| `handler` (HTTP) | parse/validate input, **authorize**, call service/operation, render response | hold business rules |
| `operation/<verb>` | multi-step business behavior in a transaction | live in a giant service file |
| `service` | simple single-step CRUD | perform authorization |
| `repository` | writes (INSERT/UPDATE/DELETE) | contain business rules |
| `queries` / `search` | reads (SELECT) | mutate |
| `policy` | answer "can this actor do X?" | touch the database |
| `decisions` (in operations) | **pure** business rules over a snapshot | import a DB driver, HTTP, or chi |

### The operation pattern

Meaningful multi-step behavior lives in `internal/<domain>/operation/<verb>/`
rather than in large service files. Each operation is a self-contained package
that follows one data flow (see [internal/article/operation/publish](internal/article/operation/publish)):

```
Contract → Scenario → Gateway.Load → Snapshot → Decisions.Make → Decision → Gateway.Apply → Result
```

| File | Role |
|---|---|
| `contract.go` | the input (plain struct) |
| `structures.go` | `Snapshot` (facts loaded from DB) and `Decision` (the write plan) — plain values |
| `decisions.go` | **pure** business rules; takes a `Snapshot` (+ `now`), returns a `Decision` or error. No I/O. |
| `gateway.go` | all DB reads/writes, bound to one transaction; translates rows ↔ operation data |
| `scenario.go` | orchestration: begin tx → load → decide → apply → commit |

Because `decisions.go` is pure, it is unit-tested with **no database**
(see [decisions_test.go](internal/article/operation/publish/decisions_test.go)).

## Authentication

Classic JWT with refresh-token rotation:

1. `POST /api/auth/register` → creates the account (bcrypt-hashed password).
2. `POST /api/auth/login` → returns an **access token** in the body and sets a
   **refresh token** as an httpOnly cookie. The refresh token's SHA-256 hash is
   stored in `refresh_tokens` so it can be revoked.
3. Send `Authorization: Bearer <access>` on subsequent requests.
4. `POST /api/auth/refresh` → validates the refresh cookie, **revokes it, and
   issues a new pair** (rotation). Reusing a rotated token fails.
5. `POST /api/auth/logout` → revokes the refresh token and clears the cookie.

Access tokens are short-lived (15m default); they carry the user id, role and
email, so authorization needs **no database lookup per request**.

## Authorization

Two complementary mechanisms, both applied **only at the handler**:

- **RBAC** — coarse, app-wide roles (`user`, `admin`). Gate whole routes with the
  `authz.RequireRole` middleware, or check `actor.IsAdmin()`.
- **Policies** — fine-grained, per-record rules. Each domain has a `Policy` with
  `Can<Action>() bool` methods (e.g. owner-or-admin can edit). The handler calls
  `authz.Authorize(policy.CanEdit())`, which returns a `403` when denied.

Every list/detail response includes a `permissions` map (`{"edit": true, ...}`)
computed from the policy, so the **frontend can show/hide actions** without
re-implementing the rules. This mirrors the source FastAPI project's
`get_permissions` behavior.

## Getting started

### With Docker (recommended)

```bash
cp .env.example .env          # adjust secrets
make docker-up                # builds images, runs migrations, starts the API
```

API: <http://localhost:3000> · health: <http://localhost:3000/health>

### Locally

```bash
cp .env.example .env
# start only Postgres (compose) or point DB_* at your own instance
docker compose up -d db
DB_HOST=localhost make migrate-up
make run
```

### Smoke test

```bash
# register + login
curl -s localhost:3000/api/auth/register -H 'Content-Type: application/json' \
  -d '{"email":"a@b.com","password":"password123","first_name":"Ada"}'

TOKEN=$(curl -s localhost:3000/api/auth/login -H 'Content-Type: application/json' \
  -d '{"email":"a@b.com","password":"password123"}' | jq -r .access_token)

# create + publish an article
curl -s localhost:3000/api/v1/articles -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' -d '{"title":"Hello","content":"World"}'

curl -s -X POST localhost:3000/api/v1/articles/1/publish -H "Authorization: Bearer $TOKEN"
```

```bash
make help            # list all targets
make run             # run the API locally
make test            # run all tests (incl. architecture guardrails)

make sqlc            # regenerate type-safe DB code after editing db/queries
make migrate-up      # apply migrations
make migrate-create name=add_widgets
make docker-up       # full stack via docker compose
```

## How to add a new domain

1. Add the table in `db/migrations/` (goose), and queries in `db/queries/`.
2. `make sqlc` to regenerate `internal/platform/database/gen`.
3. Create `internal/<domain>/` with `model.go`, `repository.go` (writes),
   `queries.go` (reads), `policy.go`, `dto.go`, `service.go`, `handler.go`,
   `routes.go`.
4. For multi-step behavior, add `internal/<domain>/operation/<verb>/`
   (contract / structures / decisions / gateway / scenario) — copy the
   `article/operation/publish` package as a starting point.
5. Mount the routes in [internal/app/server.go](internal/app/server.go).

The guardrail tests in [internal/arch](internal/arch) will keep the new code
inside the layer boundaries.

## Mapping from the FastAPI project

This template is a direct translation of a FastAPI/SQLAlchemy backend:

| FastAPI / Python | Here (Go) |
|---|---|
| `api/v1/*.py` routes | `internal/<domain>/handler.go` + `routes.go` |
| `operations/<verb>/` | `internal/<domain>/operation/<verb>/` |
| `modules/<d>/` (model, repository, queries, policy, state machine) | `internal/<domain>/` |
| `core/config.py` (pydantic-settings) | `internal/platform/config` |
| `core/db` (async engine/session) | `internal/platform/database` (pgxpool) |
| `core/permissions` (Pundit facade) | `internal/platform/authz` + per-domain `policy.go` |
| `core/schemas/base.py` envelopes | `internal/platform/httpx` |
| `exceptions/AppException` (+ `message_key`) | `pkg/apperror` |
| fastapi-users JWT | `internal/auth` (golang-jwt + bcrypt) |
| `tests/unit/test_architecture.py` | `internal/arch` |
| Alembic | goose (`db/migrations`) |
